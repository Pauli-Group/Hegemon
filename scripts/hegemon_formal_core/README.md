# Hegemon Formal Core Checker

This standalone Cargo project backs `../check_formal_core.sh`. It is intentionally not a root workspace member so it does not expand the normal Hegemon build or the native backend review package surface.

Run from the repository root:

```bash
bash scripts/check_formal_core.sh
```

The checker validates the formal/security claims ledger, validates the blueprint DAG in `config/formal-security-blueprint.json`, verifies independent bridge message vectors, checks `config/system-model-assumption-gates.json` for fail-closed release-blocking DA/storage/privacy/release/scanner/performance monitoring evidence, and checks that the active TLA+ and Lean model files are present. For every `lean_theorem` claim, it also requires a non-empty `lean_theorems` list of fully qualified theorem symbols, then resolves each symbol against non-comment theorem declarations in non-generator Lean evidence files. A claim cannot pass by pointing only at generated vectors, prose, or theorem-looking text inside Lean comments. The shell wrapper also builds the pinned Lean proof kernel through `scripts/check_lean_formal.sh`, runs the standalone runtime-environment auditor with contributor extensions and plugins disabled, calls `Lean.collectAxioms` for every claimed theorem, and checks strict JSON dependencies against `config/lean-axiom-waivers.json`, checks generated Lean-to-Rust bridge vectors through the `protocol-kernel` test target, checks generated Lean-to-Rust shielded-nullifier vectors through the `protocol-shielded-pool` test target, checks generated native action-request projection vectors at used before replay/import mutation, checks generated native block replay-refinement vectors and their staged `expected_trace` through the `hegemon-node` conformance target over the production helper composition, checks generated native storage-durability admission vectors against the production explicit flush barrier, checks generated native atomic-commit manifest admission vectors against the The axiom audit permits only declared Lean kernel axioms; unwaived local/synthetic axioms fail the gate.

The independent bridge-vector code must not depend on `protocol-kernel`; it reimplements the small canonical encoding and BLAKE3-XOF-48 derivations so production/reference drift is visible. The separate Lean conformance vector paths intentionally run inside production crate test targets and compare bridge encoding/replay plus shielded nullifier behavior against examples emitted by `formal/lean`.

The blueprint DAG is a JSON review map for the claims ledger. It rejects missing nodes, dangling edges, self-dependencies, dependency cycles, missing implementation or evidence paths, and production-eligible claims without a recorded pending/blocked target-review entry or cheap falsification cases. Target-review digests detect source drift but do not authorize or accept a review. Rust module inventories are parsed with `syn`, recursively resolve exactly one ordinary `name.rs` or `name/mod.rs` file per production module, exclude only exact `#[cfg(test)]` modules, and reject `include!` source injection, `path`, `cfg_attr`, or other production `cfg` selection; semantic binding analysis and version-2 review hashing consume the same bounded inventory. Nodes may also declare `implementation_bindings`; the checker parses the listed Rust source set, sanitizes comments plus ordinary/raw byte and C strings, rejects ambiguous duplicate free functions or same-impl methods, and verifies that the named callee is defined in non-`cfg` production code and that every required non-test caller body contains a statically live actual call expression for it, not just a same-named item reference. Required callers also fail if the body locally shadows the bound callee name through a `let`, `for`, nested `fn`, callable `const`/`static`, tuple-struct constructors, `use`, or closure-parameter binding before call scanning. Bound callee calls must be unqualified direct calls, except same-impl `self.callee(...)` and `Self::callee(...)` calls are accepted when the caller's impl defines that non-test method. Caller selectors in `required_callers` and `call_order_constraints[].caller` may be bare function names such as `verify_block_artifact` or method-qualified names such as `ReceiptRootVerifier::verify_block_artifact`; bare names match every same-named free function and impl method, and every matching non-test body must satisfy the binding, while qualified names only match methods scanned inside `impl TypeName { ... }` or `impl Trait for TypeName { ... }` blocks. Implementation bindings also reject obvious circular evidence: the bound callee cannot be the same function or method leaf as any required caller, the bound callee's non-test body cannot directly call a required caller, and order/dominance successor lists cannot name the bound callee itself as successor work. Bindings may require `result_obligation = "must_propagate_result"` so fallible gates must be propagated with `?` on the direct helper result, including turbofish calls, or an allowed simple `.map(...)`/`.map_err(...)`/`.context(...)`/`.with_context(...)` chain, returned through an explicit `return helper(...).map_err(...);`, or returned as the outer function-body tail expression. Error-swallowing combinators, assignment, semicolon discard, and nested block tails followed by later work do not satisfy that obligation. Bindings may require `result_obligation = "must_check_result_fail_closed"` so recognized fail-closed `.is_err()`, `if let Err`, direct or assignment-bound `match ... { Err(_) => ... return ... }`, immediately bound `let result = helper(...); if result.is_err() { return ... }`, immediately bound `let result = helper(...); if let Err(_) = result { return ... }`, or immediately bound `let result = helper(...); match result { Err(_) => ... return ... }` branches must hit a syntactically rejecting top-level terminal return before the surrounding statement continues; accepted shapes are empty `return`, `None`, `false`, `Err(...)`, `Err { ... }`, `ControlFlow::Break(...)`, `Ok(None)`, `Ok(false)`, or an exact two-component `(trace_identifier, Err(...))` tuple, while success-shaped returns and constructor recovery chains reject. `NativeNode::submit_action` uses the separate `must_return_submit_action_rejection` obligation for an exact `serde_json::json!` rejection with `success: false`, a null hash, and the error string; bare or locally shadowed macro names receive no credit. Bindings may also require `result_obligation = "must_check_result_loop_skip_fail_closed"` so direct `if let Err(_) = helper(...) { ... continue; }` branches or `match helper(...) { Err(_) => ... continue, ... }` arms inside loops must skip the rejected item before accept-side successor work. Bindings may require `result_obligation = "must_filter_ok_result"` so a fallible admission helper inside an iterator `filter` predicate must be the predicate's returned direct `.is_ok()` expression, and that filtered iterator must feed a terminal consumer such as `collect` or `count` whose result is returned, explicitly returned, or bound to a non-underscore local whose exact binding reaches the final block sink without rebinding or assignment; arbitrary `.map(...)` chains are not treated as preserving that filtered set. `must_contribute_to_mineable_selection` separately requires the exact admitted-transfer count, candidate-hash derivation, and final admitted-filter dataflow used by `select_mineable_actions`. Bindings may require `result_obligation = "must_return_tuple_result_component"` so tuple-returning wrappers must tail-return the second result component from a direct `let (..., result) = helper(...); result` binding. Bindings may require `result_obligation = "must_reset_work_template_on_err"` so native work-template supply advancement must use the exact empty-template fallback on `Err`: reset actions to `Vec::new()`, reset state/nullifier roots to the parent roots, reset the extrinsics root to `actions_extrinsics_root(&[])`, reset `tx_count` to `0`, and return `best.supply_digest`. Bindings may require `result_obligation = "must_guard_false_fail_closed"` so boolean predicates in `if !helper(...) { return ... }` guards must top-level terminal-return before successor work on false. Bindings may require `result_obligation = "must_match_some_or_return_supply_delta_invalid"` for native replay expected-supply reconstruction, or `result_obligation = "must_match_some_and_compare_supply_claim"` for generic PoW checked expected-supply/header-claim comparison. Every call site for the bound helper in a required caller must satisfy the declared obligation. Bindings may also declare `call_order_constraints`, which require the named callee call to occur before each declared successor call in the non-test caller body. Successors in `callee_must_precede` may be bare function identifiers such as `persist_block` or conservative path-qualified call selectors such as `RwLock::new`, `Arc::new`, or `self.persist_block`; bare identifiers keep the broad match against free-function and method calls, while qualified selectors require the same receiver/type path immediately before the call. Constraints with `must_dominate_successors` additionally require the callee call to sit in an earlier statement in the same Rust block or an earlier ancestor-block statement than every successor call, rejecting sibling-branch order spoofing. This is intentionally a narrow production-wiring gate, not a Rust verifier: it prevents critical Lean-modeled helpers from becoming test-only, locally shadowed, silently disconnected from named import/replay paths, ignored after returning an error, moved after declared mutation/persistence/publication boundaries, or wired through a helper/caller cycle that can satisfy the blueprint without independent implementation evidence.
