# Hegemon Formal Core Checker

This standalone Cargo project backs `../check_formal_core.sh`. It is intentionally not a root workspace member so it does not expand the normal Hegemon build or the native backend review package surface.

Run from the repository root:

```bash
bash scripts/check_formal_core.sh
```

The checker validates the formal/security claims ledger, validates the blueprint DAG in `config/formal-security-blueprint.json`, verifies independent bridge message vectors, and checks that the active TLA+ and Lean model files are present. For every `lean_theorem` claim, it also requires a non-empty `lean_theorems` list of fully qualified theorem symbols, then resolves each symbol against non-comment theorem declarations in non-generator Lean evidence files. A claim cannot pass by pointing only at generated vectors, prose, or theorem-looking text inside Lean comments. The shell wrapper also builds the pinned Lean proof kernel through `scripts/check_lean_formal.sh`, audits `#print axioms` output for every claimed theorem against `config/lean-axiom-waivers.json`, checks generated Lean-to-Rust bridge vectors through the `protocol-kernel` test target, checks generated Lean-to-Rust shielded-nullifier vectors through the `protocol-shielded-pool` test target, checks generated native action-state effect vectors through the `hegemon-node` planner used before replay/import mutation, checks generated native block replay-refinement vectors and their staged `expected_trace` through the `hegemon-node` conformance target over the production helper composition, runs the announced-block replay-before-payload regression, checks declared Rust implementation bindings and call-order constraints for critical supply/replay/import helpers, checks the generated native RISC Zero release-verifier fail-closed vectors through the `hegemon-node` test target, checks generated release PQ-binary policy vectors through the security-audit policy helper, checks generated dependency-audit waiver policy vectors through the dependency-audit policy helper, checks generated native backend review-policy vectors plus the actual `testdata/native_backend_vectors/bundle.json`, checks generated native backend release-posture vectors plus the packaged `audits/native-backend-128b/native-backend-128b-review-package.tar.gz`, runs the existing `native-backend-ref` vector verifier for native tx-leaf and receipt-root proof artifacts, and checks that the packaged native backend release posture remains candidate-under-review until an external acceptance artifact deliberately changes that policy. The axiom audit permits only declared Lean kernel axioms; unwaived local/synthetic axioms fail the gate.

The independent bridge-vector code must not depend on `protocol-kernel`; it reimplements the small canonical encoding and BLAKE3-XOF-48 derivations so production/reference drift is visible. The separate Lean conformance vector paths intentionally run inside production crate test targets and compare bridge encoding/replay plus shielded nullifier behavior against examples emitted by `formal/lean`.

The blueprint DAG is a JSON review map for the claims ledger. It rejects missing nodes, dangling edges, self-dependencies, dependency cycles, missing implementation or evidence paths, and production-eligible claims without accepted target review or cheap falsification cases. Nodes may also declare `implementation_bindings`; the checker parses the listed Rust source file and verifies that the named callee is defined in non-test code and that every required non-test caller body references it. Bindings may also declare `call_order_constraints`, which require the named callee to appear before each declared successor callee in the non-test caller body. This is intentionally a narrow production-wiring gate, not a Rust verifier: it prevents critical Lean-modeled helpers from becoming test-only, silently disconnected from named import/replay paths, or moved after declared mutation/persistence boundaries. The current gate includes 23 implementation bindings around native supply advancement, coinbase accounting, coinbase payload admission, transfer state admission, decoded block-action validation, block-artifact binding, block replay refinement, mined-work/PoW mutation and atomic mined-block commit ordering, canonical-index rebuild planning, and atomic canonical reorg commit ordering, plus 8 order constraints covering 20 callee-precedence edges, while full implementation equivalence remains a separate goal. Lean theorem evidence must still build under `formal/lean`; JSON metadata alone is not treated as a proof.
