# Transaction Proof Soundness Analysis

This note freezes the current Hegemon transaction-proof release profile, the exact code-derived AIR terms used to analyze it, and the measured proof-size sweep at fixed `log_blowup = 4`.

It is intentionally narrow. This is not a full theorem for every non-interactive Fiat-Shamir detail of the shipped Goldilocks/Poseidon2/FRI stack. It is the exact engineering note needed to answer three concrete questions:

1. What release FRI profile does the current product actually bind to?
2. What exact algebraic AIR error terms can be derived directly from code for that profile?
3. How much proof-size reduction is available from lower query counts before the current Hegemon `128-bit` engineering floor is violated?

The exact algebraic error numerators currently derived from code are:

- Alpha-combination numerator: `250`

Those map to these conservative floors:

- Alpha-combination floor: `120` bits
- Out-of-domain floor: `111` bits
- Algebraic union floor: `111` bits

The important point is that the algebraic AIR part is not the dominant release risk at the current profile. The dominant release trade-off is the FRI/query layer.

## Current engineering rule

Hegemon’s current release rule is still the engineering heuristic used throughout the repo:

`effective_log_blowup * num_queries + query_pow_bits`

At the current release profile, that is `4 * 32 + 0 = 128`.

This note does not upgrade that rule into a full theorem for non-interactive FRI. The current state of the literature makes that distinction matter. The concrete-security work by Block and Tiwari shows that deployed FRI systems often rely on conjectured rather than fully provable non-interactive margins, and that parameter choices can look much stronger under conjectured analysis than under strict provable bounds.

Primary source:

- Alexander R. Block and Pratyush Ranjan Tiwari, “On the Concrete Security of Non-interactive FRI,” ePrint 2024/1161, https://eprint.iacr.org/2024/1161.pdf

So the correct interpretation is:

- the repo now has an exact code-derived AIR analysis,
- the repo still uses an engineering FRI rule for the release target,
- and lower-query profiles must clear that rule before they can be called release-safe.

## Exact fixed-`log_blowup = 4` sweep

The checked-in artifact is [tx_proof_profile_sweep.json](/Users/pldd/Projects/Reflexivity/Hegemon/docs/crypto/tx_proof_profile_sweep.json). It is generated by:

    cargo run -p transaction-circuit --bin tx_proof_profile_sweep

The current measured rows are:

- `32` queries: `354081` bytes, shrink `1.000x`, heuristic FRI bits `128`
- `28` queries: `310631` bytes, shrink `1.140x`, heuristic FRI bits `112`
- `24` queries: `267031` bytes, shrink `1.326x`, heuristic FRI bits `96`
- `20` queries: `223474` bytes, shrink `1.584x`, heuristic FRI bits `80`
- `16` queries: `179934` bytes, shrink `1.968x`, heuristic FRI bits `64`

The exact conclusion is straightforward:

- `16` queries gives almost a `2x` proof-size reduction on the exact current transaction circuit.
- `16` queries does not satisfy the current Hegemon `128-bit` engineering rule.
- `28`, `24`, and `20` queries also fail that current rule.

So the current release answer is negative: the present codebase does not justify a release move from `32` queries to `16` queries at fixed `log_blowup = 4`.

## Implication for the next optimization step

Because the exact algebraic AIR floor is still `111` bits while the release engineering rule is dominated by the FRI/query term, the next serious proof-size work should target the opening/query layer instead of pretending that a query cut is “free.”

That is why the next engineering spike is documented separately in [tx_proof_size_reduction_paths.md](/Users/pldd/Projects/Reflexivity/Hegemon/docs/crypto/tx_proof_size_reduction_paths.md).
