# Hegemon Lean Formal Core

This directory contains the machine-checked Lean 4 layer for Hegemon's production-critical validity rules.

The project is pinned by `lean-toolchain` and builds with:

```bash
bash ../../scripts/check_lean_formal.sh
```

Current proved kernel:

- `Hegemon/Bytes.lean` defines shared executable byte, endian, hex, and deterministic patterned-byte helpers used by generated Lean conformance vectors.
- `Hegemon.Bridge.accept_prevents_duplicate` in `Hegemon/Bridge/Replay.lean` proves that once the executable inbound replay-state transition accepts a replay key and returns a next state, the same key cannot be accepted again from that next state.
- `Hegemon.Bridge.stage_prevents_duplicate_pending`, `Hegemon.Bridge.import_prevents_reimport`, and `Hegemon.Bridge.import_prevents_restaging` prove the two-phase inbound replay behavior used by the native bridge path: a staged key cannot be staged twice, and an imported key cannot be imported or staged again.
- `Hegemon/Bridge/Encoding.lean` defines the executable bridge byte-encoding grammar for `BridgeMessageV1`. `Hegemon/Bridge/GenerateVectors.lean` emits Lean-derived bridge encoding and replay examples, and `bash ../../scripts/check_formal_core.sh` compares them against production Rust helpers.
- `Hegemon.Bridge.MessageRoot.empty_valid`, `Hegemon.Bridge.MessageRoot.ordered_pair_valid`, `Hegemon.Bridge.MessageRoot.short_hash_rejected`, `Hegemon.Bridge.MessageRoot.long_hash_rejected`, `Hegemon.Bridge.MessageRoot.ordered_pair_transcript`, `Hegemon.Bridge.MessageRoot.ordered_pair_differs_from_reversed`, and `Hegemon.Bridge.MessageRoot.count_binds_prefix` prove representative facts about the executable bridge message-root transcript grammar. `Hegemon/Bridge/GenerateVectors.lean` emits Lean-derived transcript bytes, and `bash ../../scripts/check_formal_core.sh` compares them against the production `protocol-kernel` root preimage helper.
- `Hegemon.Bridge.LongRange.valid_shape_accepts`, `Hegemon.Bridge.LongRange.rejects_bad_verifier_hash`, `Hegemon.Bridge.LongRange.rejects_message_count_mismatch`, `Hegemon.Bridge.LongRange.rejects_tip_mmr_len_mismatch`, `Hegemon.Bridge.LongRange.rejects_tip_not_after_message`, `Hegemon.Bridge.LongRange.rejects_message_opening_leaf_mismatch`, `Hegemon.Bridge.LongRange.rejects_message_index_oob`, `Hegemon.Bridge.LongRange.rejects_sample_height_mismatch`, `Hegemon.Bridge.LongRange.rejects_under_confirmed`, `Hegemon.Bridge.LongRange.rejects_insufficient_tip_work`, and companion rejection theorems prove representative facts about the executable bridge long-range proof-shape admission table. `Hegemon/Bridge/GenerateLongRangeVectors.lean` emits Lean-derived accept/reject examples, and `bash ../../scripts/check_formal_core.sh` compares them against the production `consensus-light-client` helper called by the long-range verifier.
- `Hegemon.Consensus.higher_work_wins`, `Hegemon.Consensus.lower_work_loses`, `Hegemon.Consensus.equal_work_higher_height_wins`, `Hegemon.Consensus.equal_work_lower_height_loses`, `Hegemon.Consensus.equal_work_height_uses_hash_order`, and `Hegemon.Consensus.same_tip_not_better` prove deterministic two-tip PoW fork-choice ordering: higher work wins, then higher height, then lower hash. `Hegemon/Consensus/GenerateVectors.lean` emits Lean-derived fork-choice examples, and `bash ../../scripts/check_formal_core.sh` compares them against the production `consensus::fork_choice` helper used by `PowConsensus` and native block import.

This is real Lean theorem and conformance-vector evidence, but it is deliberately narrow. It does not prove BLAKE3 cryptographic security or implementation equivalence for bridge roots/replay keys, BLAKE2 action-order key derivation, complete bridge light-client soundness, external-chain covenant behavior, header hashing, monetary policy economics, retarget economics, recursive proof cryptographic soundness, native lattice backend security, native tx-leaf proof soundness, native receipt-root cryptographic fold soundness, full network/finality behavior, note commitment correctness, Poseidon2 Merkle hash security or implementation equivalence, nullifier derivation, ciphertext hash derivation, BLAKE3 statement-hash collision resistance or implementation equivalence, full AIR/proof-system soundness, or full native-node equivalence.
